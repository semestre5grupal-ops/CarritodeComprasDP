2.2.6. ARIA: cuándo sí y cuándo no
ARIA no reemplaza HTML. Primero elige el elemento correcto; luego, si no
existe equivalente
nativo, usa ARIA para completar:
• Varias nav → diferéncialas con aria-label="Principal", arialabel="Breadcrumbs".
• Acordeón custom (si no usas details): role="button", aria-expanded, ariacontrols, y
teclas Enter/Espacio.
• Evita roles redundantes: no pongas role="navigation" en nav, ni role="main"
en main.
ARIA es para complementar a HTML no para sustituirlo

2.4.2. Perceptible: que la información “llegue”
Imágenes. Si la imagen comunica algo, pon alt descriptivo; si es decorativa,
alt="". Si requiere
contexto, usa figure + figcaption.
2.4.3. Operable: teclado first
El sitio debe poder usarse completamente con teclado. Asegura foco visible y
orden lógico. Modales accesibles (idea mínima). Deben: 1) atrapar foco, 2)
anunciarse, 3) cerrarse con Esc, 4)
devolver el foco al disparador.
2.4.4. Comprensible: que se entienda lo que pides
Usa lenguaje claro y etiquetas reales. Las ayudas y errores deben estar
vinculados y anunciarse. Mantén jerarquía de encabezados (h1→h2→h3…) y
patrones de navegación consistentes entre
páginas (mismo lugar para menú, buscador, etc.).
2.4.5. Robusto: que funcione “a prueba de futuro”
Semántica nativa primero: header, nav, main, section, article, aside, footer,
figure/figcaption,
time, dl/dt/dd, details/summary.
ARIA solo cuando falte semántica nativa o para desambiguar (ej.: varias nav
con aria-label).
Valida HTML/CSS, prueba en más de un navegador y, si puedes, con un lector
